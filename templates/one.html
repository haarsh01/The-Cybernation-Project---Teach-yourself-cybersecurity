#1 Injection

Injection or SQL injection is a technique that can be used by malicious external entities to send invalid data to the web applications to make them malfunction and do something they were not intentionally intended to do. This vulnerability is most often associated with those web applications that are linked to a database in the backend. This encompasses almost the entire spectrum of online applications as almost all of them are data driven these days. The injection attack most commonly happens when SQL queries which are generated by the website to access the abstract database are tampered with and made to consume untrusted data. This situation arises when the data consumed by the web applications is not validated or sanitized.

This is arguably one of the most commonly used web application attack techniques. All those website operations that accept parameters as inputs are vulnerable to the SQL injection attack. The database query that is to be sent by the application to the remote or cloud-based database is carefully manipulated and recrafted into a malicious statement. To understand how this technique is used to launch attacks on web applications it is important to understand that whenever any information is fetched or entered into the system, a SQL query is generated. During the injection attack, it is this generated query that is tampered with. It is also important to understand that changes can only be made to the input that is given to the application and not the underlying query itself as the hacker does not, in this case, has access to the website’s source code. When the query evaluates to true, the data is suitably fetched and displayed. If it evaluates to false, an error is thrown.

The input is manipulated in a manner so that the resultant query is altered, and the malicious query thus generated can fetch the desired results to the hacker. One of the most commonly used techniques in this case is the usage of the OR logic gate function. This function and the effect it has on the query is illustrated in the figure below:

Original Query Malacious Query
Select * from users
Where
username=’abc’
And
password=’123’
Select * from users
Where
username=’’OR 1=1--'
And
password=’123’
Figure 1: The OR function to launch Injection attack

In the example in the above illustration, the web application prompts the user to enter the username and password. If the username entered and the corresponding password are found in the database by the generated query, it evaluates to true and the data is fetched. Otherwise an error is thrown. When the OR logic function is used along with the condition 1=1, the query always evaluates to true since, 1=1 is always true. Moreover, the double dashes and quotation mark at the end of the malicious statement comment out the rest of the query thereby making the password matching obsolete for that cycle. Thus, in this case, a true flag is generated as the result of the query and the data is returned by database and displayed without the proper credentials making the hack successful. The injection technique is most commonly used to control application behavior based on the data in the database, for instance tricking an application into allowing a login without a valid password. This attack may also be used to alter data in the database without authorization, for example by creating fraudulent records and adding users or promoting existing users to higher access levels. This may also be used to delete data from the database. Another application is in accessing data without authorization, for instance by tricking the database into providing too many results for a given query.

Some of the suggestions that can be given to developers to prevent SQL injection attacks are: Escaping all user supplied input: This helps to circumvent inputs where the user enters specific characters or words that may be malicious in nature and have risks associated with the. Escaping a character tells the database not to parse it as a command but instead to treat is as a literal input. Enforcing least privilege: Herein, the exposure to injection vulnerability is reduced by limiting permissions to the narrowest scope required to execute the relevant query. In other words, this means that an administrative account should not be allowed to execute SQL commands as a result of an API call from an unauthorized request.
