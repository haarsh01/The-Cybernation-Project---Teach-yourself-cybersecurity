#7 Cross-Site Scripting (XSS)

Cross Site Scripting (XSS) is a widespread vulnerability that affects many web applications. XSS attacks consist of injecting malicious client-side scripts into a website and using the website as a propagation method. It can also be defined as a code injection attack that is executed on client side of the web application which in most of the cases is the victim’s web browser. In XXS vulnerability, the attacker injects malicious code into the browser to make the application do something it wasn’t originally intended to do. Exploiting this vulnerability, the hacker may steal cookies, session tokens and other sensitive information. This type of vulnerability may also be used to modify the original contents of the website therefore it is as harmful to the user of the application as it is to the affiliated organisation itself.

In cross-site scripting attacks, the hacker tampers with the web application and steals credentials from the browsers or sometimes even via the servers. There can mainly be three types of XXS attacks. The first one is the Reflected cross site scripting attack. In this case, the malicious script is executed on the victim side and is not stored on the server. The second type of XXS attack is termed as the Stored XXS and in this case, the malicious script is stored and executed on the server. This script is executed every time the site is requested by the client/victim. The final type of XXS attack is the DOM or Document object model attack. This type of attack is a client side attack and the script is not stored on the server. In this case, the legitimate script as requested by the server is executed first followed by the malicious one.

Cross site scripting can be prevented if the following suggestions are incorporated into a given web application: Escaping the user input and thereby taking away the special features of certain characters. This prevents injection of html tags and other symbols utilized to create malicious scripts that may be fed into the application. All the inputs should be treated as threats and the data entered should always be validated and sanitisation of inputs should be done religiously. This allows the elimination of script tags and usage of regular expressions. The output should always be encoded. The use of right response headers may also be done to closely govern the kind of data that is expected and displayed by the application. The use of content security policy may also be done.
